# SignPath status

These files describe the intended SignPath configuration for QSDM Hive.

The SignPath Foundation application was reviewed and declined for now because
QSDM does not yet have enough external public trust signals: adoption, stars,
forks, contributors, independent articles, community discussions, or
institutional backing. This is not a technical rejection of QSDM; it means the
Foundation certificate is unavailable until the project has broader public
visibility.

Current release posture:

- Windows Hive releases are unsigned by Microsoft Authenticode.
- Users must verify SHA-256 checksums from `qsdm.tech/downloads`.
- Windows SmartScreen may warn because the publisher has no established
  Authenticode reputation.
- QSDM Hive 1.3.96 authenticates release manifests and installer hashes with a
  pinned ML-DSA-87 project key, but that does not replace Microsoft
  Authenticode trust.
- The optional GitHub `sign-production` job runs only when the repository
  variable `SIGNPATH_ENABLED` is exactly `true`.
- Paid SignPath/Authenticode or a future accepted Foundation reapplication
  remains the path to Windows verified-publisher UX.

## Intended project configuration

- Project slug: `qsdm-hive`
- Unpacked artifact configuration slug: `hive-unpacked-v1`
- Installer artifact configuration slug: `hive-installer-v1`
- Production signing policy: manual approval by the repository owner
- Trusted build system: GitHub connector for `blackbeardONE/QSDM`
- Allowed runner: GitHub-hosted Windows runner only

## Account setup if SignPath is approved later

1. Enable multi-factor authentication for the SignPath and GitHub accounts.
2. Install the SignPath GitHub App for `blackbeardONE/QSDM`, add the predefined
   GitHub.com trusted build system, and link it to the `qsdm-hive` project.
3. Create a CI submitter that cannot approve requests. Store only its token as
   the `SIGNPATH_API_TOKEN` GitHub Actions secret.
4. Store `SIGNPATH_ORGANIZATION_ID`, `SIGNPATH_PROJECT_SLUG`, and
   `SIGNPATH_SIGNING_POLICY_SLUG` as repository variables. Use the intended
   slugs above unless SignPath assigned different values. Set
   `SIGNPATH_ENABLED=true` only after all credentials and policy controls are
   ready.
5. Upload the two XML files in this directory as the named artifact
   configurations and inspect their resolved artifact trees.
6. Configure the production policy for manual approval and configure the
   `signpath-production` GitHub environment with deployment protection.
7. Generate a `hive-v<package-version>` tag only from a commit already merged
   into `main`. Never put an approver credential or a certificate private key
   in GitHub Actions.

The first request opens the GitHub artifact ZIP, then signs the QSDM-owned
executables in its nested ZIP containing `win-unpacked`. The signed directory
is then used as electron-builder's `--prepackaged` input. The resulting NSIS
installer is submitted through the second configuration. This two-stage flow
is required because NSIS is not a SignPath deep-signing container: signing
only the outer installer would leave its installed QSDM executables unsigned.

After packaging, the workflow extracts the seven QSDM executables from NSIS
and compares their SHA-256 hashes with the signed source payload. It then
verifies the outer installer signature before publishing release evidence.

Upload both XML files to the corresponding artifact configurations after
SignPath creates the project. Do not add SignPath API tokens to source control.
Store the API token as a GitHub Actions secret and the organization/project/
policy slugs as repository variables. Install the SignPath GitHub App for the
repository so origin metadata cannot be supplied by the build script.

## First Authenticode-signed release

Existing development builds identify their configured publisher as `QSDM` or
are unsigned. The first installer signed by a recognized Authenticode
certificate is therefore a publisher transition and must be downloaded and
installed manually after its signature and checksum are verified. Do not
advertise that transition as an automatic update. Once users run an
Authenticode-signed Hive build, subsequent releases signed by the same trusted
identity can use the normal signed updater path.

Before production use, upload one unsigned sample generated by
`QSDM/deploy/scripts/prepare_hive_signpath_payload.ps1` and review SignPath's
resolved artifact tree. It must sign only the seven explicitly listed QSDM
executables. Upstream Electron/Chromium files must retain their upstream
signatures and must not receive the QSDM project signature.

See [CODE_SIGNING_POLICY.md](../CODE_SIGNING_POLICY.md) for the public policy
and [BUILD_AND_RELEASE_GUIDELINES.md](../QSDM/docs/docs/BUILD_AND_RELEASE_GUIDELINES.md)
for the complete release gates.

## Reapply criteria

Before reapplying for Foundation signing, collect objective trust signals:

- GitHub stars, forks, issues, and external contributors.
- Public release history with checksums and reproducible release evidence.
- Independent articles, videos, forum threads, or third-party references.
- Real user reports or operator deployments that are safe to cite publicly.
- A cleaner project homepage and README that explains the threat model and
  release-verification process.
