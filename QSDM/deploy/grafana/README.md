# Grafana — QSD dashboards

This directory contains two flavors of dashboards:

| Artifact | Source | Purpose |
| --- | --- | --- |
| `QSD-overview.json` | hand-curated | starter dashboard with the 17 highest-signal panels (NVIDIA-lock, NGC, transactions, network, submesh) |
| `dashboards/QSD-alerts-overview.json` + `dashboards/QSD-runbook-*.json` | auto-generated by [`scripts/gen_grafana_dashboards.py`](../../../scripts/gen_grafana_dashboards.py) | one panel per alert, organised by runbook; URLs match the `dashboard_url` annotation on each Prometheus rule |

The hand-curated dashboard gives you a "first 60 seconds at a node" view; the auto-generated dashboards give you "click here from PagerDuty and see the firing alert immediately."

## Import

1. Add a **Prometheus** data source pointing at your Prometheus server (which scrapes `deploy/prometheus/scrape_QSD.example.yml` or `prometheus.QSD.example.yml`).
2. In Grafana: **Dashboards → New → Import** → upload **`QSD-overview.json`** (or any of the per-runbook JSONs from `dashboards/`).
3. When asked for **DS_PROMETHEUS**, pick that Prometheus data source.

### File-provisioned auto-loading (recommended)

The 11 per-runbook dashboards plus the master alerts overview can be auto-loaded by Grafana via the file-provisioning shim at [`provisioning/dashboards/QSD-runbooks.example.yaml`](provisioning/dashboards/QSD-runbooks.example.yaml).

1. Copy that file to `/etc/grafana/provisioning/dashboards/QSD-runbooks.yaml` on your Grafana host.
2. Edit the `options.path:` line to point at wherever you copied the contents of `QSD/deploy/grafana/dashboards/`.
3. Restart Grafana once. Subsequent edits to the JSON files (e.g. after re-running the generator) are picked up without a restart — `updateIntervalSeconds: 30` re-scans the directory every half-minute.

### Regenerating the auto dashboards

```
python scripts/gen_grafana_dashboards.py
```

The script reads `QSD/deploy/prometheus/alerts_QSD.example.yml`, groups the 38 alerts by their `runbook_url`, and emits one stat-panel-per-alert dashboard for each runbook. Panel IDs are stable across regenerations (derived from the alphabetical ordering of alertnames), so the `dashboard_url` annotation embedded in each Prometheus rule (`?viewPanel=N`) keeps pointing at the right panel even when alerts are added or removed.

### Optional: file provisioning (fixed datasource UID)

To avoid the import picker and match panel UIDs explicitly:

1. Copy **`provisioning/datasources/prometheus.example.yml`** into Grafana’s provisioning folder (e.g. `/etc/grafana/provisioning/datasources/prometheus.yml`). Edit `url:` to your Prometheus (`http://localhost:9090`, `http://prometheus:9090` in Docker, etc.).
2. The example sets datasource **`uid: prometheus`**. Either choose that datasource in the import UI, or in the dashboard JSON replace every `"uid": "${DS_PROMETHEUS}"` with `"uid": "prometheus"` and remove the top-level **`__inputs`** array before import.

### Troubleshooting

- **All panels empty:** Prometheus has no data (check scrape targets / Bearer secret on the node’s `metrics_scrape_secret`) or the wrong datasource was selected at import.
- **Strict dashboard auth:** If the node uses `strict_dashboard_auth` and JWT init failed, UI routes may return 503; scraping still works when `metrics_scrape_secret` matches.

## Metrics

Panels use the `QSD_*` series from **`GET /api/metrics/prometheus`** on the node dashboard (see `deploy/prometheus/README.md`). Includes **submesh** stats (`QSD_submesh_*`) when submesh profiles are in use.

Adjust refresh interval and time range in the UI as needed.
