# Security Incident — Operator Runbook

Runbook for the nine `QSD-security` Prometheus
alert group entries (MED-8 in
[SECURITY_AUDIT.md](../SECURITY_AUDIT.md)). All
nine alerts page the on-call security operator;
the most aggressive escalation is reserved for
**stolen-token replay**
([§3.6](#36-QSDsecuritytokenrevokedhit)), which is
the only `critical` severity in the group.

| Alert | Severity | Default `for:` | Anchor |
|---|---|---|---|
| `QSDSecurityFailedLoginBurst`             | warning  | 10m | [§3.1](#31-QSDsecurityfailedloginburst) |
| `QSDSecurityAccountLockoutBurst`          | critical | 5m  | [§3.2](#32-QSDsecurityaccountlockoutburst) |
| `QSDSecurityRateLimitViolationSpike`      | warning  | 10m | [§3.3](#33-QSDsecurityratelimitviolationspike) |
| `QSDSecurityCSRFFailureSpike`             | warning  | 5m  | [§3.4](#34-QSDsecuritycsrffailurespike) |
| `QSDSecurityInvalidTokenSpike`            | warning  | 10m | [§3.5](#35-QSDsecurityinvalidtokenspike) |
| `QSDSecurityTokenRevokedHit`              | critical | 5m  | [§3.6](#36-QSDsecuritytokenrevokedhit) |
| `QSDSecurityRequestSignatureFailureSpike` | warning  | 5m  | [§3.7](#37-QSDsecurityrequestsignaturefailurespike) |
| `QSDSecurityRequestTimeoutSpike`          | warning  | 10m | [§3.8](#38-QSDsecurityrequesttimeoutspike) |
| `QSDSecurityCORSRejectionSpike`           | warning  | 10m | [§3.9](#39-QSDsecuritycorsrejectionspike) |

> **Why a single runbook for nine alerts?** The
> `QSD_security_*` counters are a single
> coherent surface — every metric is recorded
> from a middleware in the chain documented in
> `pkg/api/server.go::setupMiddleware` (outermost
> first: `SecurityHeaders → CORS → AuditLog →
> Deprecation → RequestTimeout → RateLimit →
> CSRF → RequestSigning → Auth → AdminAccess`).
> Bundling them keeps the triage flow consistent:
> the operator first identifies which middleware
> the alert points at, then drills into that
> layer's evidence.

---

## 1. Glossary (60-second skim)

- **Middleware chain** — ordered list of
  `func(http.Handler) http.Handler` wrappers in
  `pkg/api/server.go::setupMiddleware`. Every
  inbound request flows through all of them in
  the listed order; each emits one or more
  `QSD_security_*` counters on rejection.
- **Audit log** — JSON-lines log written by
  `AuditLogMiddleware` (the outermost layer
  after `CORS`). Contains every accepted AND
  rejected request, keyed by an opaque
  per-request `request_id`. The triage flow in
  every §3 mode below assumes you can `grep` it
  by IP, user, or path.
- **Correlation ID** — six-byte hex prefix
  generated by `pkg/api/error_sanitize.go::WriteServerError`
  and appended to every 5xx response in
  production mode. Lets you cross-reference a
  user's "I got an error with id=ab12cd34" report
  to the structured log line that captured the
  raw error.
- **CSRF token** — opaque 256-bit nonce minted by
  `GET /api/v1/csrf-token` and required (header
  `X-CSRF-Token` + cookie `QSD_csrf`, must
  match) on every state-changing request from a
  cookie-authenticated browser. Bearer-token
  callers (SDKs) bypass CSRF.
- **JWT nonce** — random claim baked into every
  issued JWT, used as the key for
  `TokenRevocationStore`. Lets `Logout` revoke
  a specific token instance without invalidating
  the user's other concurrent sessions.

## 2. First-90-seconds checklist

The same five checks apply to all nine alerts —
do them before reaching for the per-alert
section.

1. **Confirm the alert is real, not a stuck
   counter.** Re-run the alert's `expr` directly
   in Prometheus; if the rate is back to
   baseline, the alert is decaying and the
   incident is over.
2. **Identify the source IP distribution.** Tail
   the audit log for the relevant path / status,
   then `awk` on `client_ip` and count. A single
   `/32` dominating is almost always a script;
   a wide `/16` distribution is broad probing.
3. **Check the WAF / LB for an upstream block
   list.** If the operator runs an upstream
   gatekeeper (Cloudflare, nginx ratelimit), the
   first lever is "block at the edge" — cheaper
   than escalating to a node-level config
   change.
4. **Page the on-call security operator if
   severity is `critical`.** The two `critical`
   alerts (`AccountLockoutBurst`,
   `TokenRevokedHit`) imply an attack landed
   past at least one layer of defence — they
   page out of band, not just into Slack.
5. **Snapshot the audit log.** Before any
   mitigation rolls the log over, capture the
   current 24h slice for forensics. The
   credential-compromise scenarios in §3.2 and
   §3.6 need the unaltered log for incident
   review.

## 3. Modes

### 3.1. QSDSecurityFailedLoginBurst

#### What triggered it

`rate(QSD_security_failed_logins_total[5m]) >
1.0` sustained for 10m.

The `Login` handler in
`pkg/api/handlers.go` calls
`monitoring.RecordFailedLogin()` on every
unknown-user / wrong-password path. >1/s
sustained for 10m is well above the
fat-fingered-password baseline and below the
rate-limit threshold — the gap is exactly the
credential-stuffing / password-spray window.

#### Triage

```
# How many distinct usernames? -> stuffing vs. spray.
jq -r 'select(.path=="/api/v1/auth/login" and .status==401) | .username' /var/log/QSD/audit.log \
  | sort -u | wc -l

# Source IP distribution.
jq -r 'select(.path=="/api/v1/auth/login" and .status==401) | .client_ip' /var/log/QSD/audit.log \
  | sort | uniq -c | sort -rn | head -10
```

#### Mitigation

- **Single target, single IP:** the LB hard-blocks the
  IP; nothing else to do.
- **Single target, distributed IPs:** tighten
  `QSD_LOCKOUT_THRESHOLD` (defaults to 5)
  to 3 for the duration. Notify the targeted
  user. The lockout layer will absorb the rest.
- **Many targets, few IPs:** WAF block, then page
  the security lead — this is the prelude to a
  large compromise.

#### Recovery validation

The alert clears when the 5m rate drops below
1.0/s. The lockout counter (§3.2) should NOT
have fired — if it did, escalate to §3.2.

### 3.2. QSDSecurityAccountLockoutBurst

#### What triggered it

`increase(QSD_security_account_lockouts_total[15m]) > 5`
sustained for 5m.

`AccountLockoutManager` only fires after the
configured consecutive-failure threshold
(`QSD_LOCKOUT_THRESHOLD`, defaults to 5).
A >5 lockout count in 15m means at least 5
distinct accounts absorbed enough wrong
passwords to lock — a strong signal of a
coordinated credential-spray or stolen-password
replay.

#### Triage

```
# Which accounts locked? Capture the full list
# before lockouts expire.
jq -r 'select(.event=="account_lockout") | .username' /var/log/QSD/audit.log

# Are the lockouts time-clustered? (paste output
# into a histogram tool — clustering implies
# coordinated attack.)
jq -r 'select(.event=="account_lockout") | .ts' /var/log/QSD/audit.log
```

#### Mitigation

1. Snapshot the audit log NOW — `cp -p audit.log
   audit-incident-$(date +%s).log`.
2. Hard-block source IP space at the LB.
3. Force password reset for every locked
   account (the password landscape leaked, even
   if the attempt failed).
4. Open an incident review ticket. This event
   warrants a post-mortem regardless of attack
   success.

#### Recovery validation

The 15m `increase()` must drop back to 0 AND
the locked accounts must be manually unlocked
or wait out the lockout window
(`QSD_LOCKOUT_DURATION`, defaults to 30m).

---

### 3.3. QSDSecurityRateLimitViolationSpike

#### What triggered it

`rate(QSD_security_rate_limit_violations_total[5m]) > 5.0`
sustained for 10m.

Per-IP and per-role limits are tripping faster
than legitimate traffic ever should.

#### Triage

```
# Which path is being hammered?
jq -r 'select(.status==429) | .path' /var/log/QSD/audit.log \
  | sort | uniq -c | sort -rn | head -10

# Who is the offender?
jq -r 'select(.status==429) | .client_ip' /var/log/QSD/audit.log \
  | sort | uniq -c | sort -rn | head -10
```

#### Mitigation

- **Single offending IP:** LB hard-block. Done.
- **Wide IP distribution, legitimate traffic
  burst (e.g. launch day):** raise the
  relevant role tier in `RoleRateLimiterConfig`
  for the duration, then revert. The alert is
  doing its job — it's flagging that the
  rate-limit is the active bottleneck.
- **Wide IP distribution, attack:** upstream
  WAF, escalate.

#### Recovery validation

5m rate must drop below 5.0/s for the alert to
clear.

### 3.4. QSDSecurityCSRFFailureSpike

#### What triggered it

`rate(QSD_security_csrf_failures_total[5m]) > 0.1`
sustained for 5m.

`CSRFMiddleware` rejected a state-changing
request because either the `X-CSRF-Token` header
was missing/mismatched against the `QSD_csrf`
cookie, OR the synchronizer-token store
rejected it (expired / not issued / user-bind
mismatch).

#### Triage

```
# Which of the three failure reasons dominates?
jq -r 'select(.event=="csrf_reject") | .reason' /var/log/QSD/audit.log \
  | sort | uniq -c | sort -rn

# Reasons (from pkg/api/csrf.go::writeCSRFError):
#   "missing X-CSRF-Token header"          -> client misconfig
#   "CSRF token mismatch between header
#    and cookie"                            -> double-submit fail
#   "CSRF validation failed: ..."          -> synchronizer store fail
```

#### Mitigation

- **Mostly "missing X-CSRF-Token":** a freshly
  deployed client skipped the
  `GET /api/v1/csrf-token` handshake. Identify
  the client by `User-Agent`, fix the
  integration. No node-side action.
- **Mostly "token mismatch":** the
  client's cookie store and JS variable
  diverged — usually a bug in the SDK's
  CSRF helper. Same fix as above.
- **Mostly "validation failed":** an active
  attack or a token-store cleanup bug. Capture
  the audit log and escalate; if the store has
  been running >24h and shows stale-token
  retention, restart the API.

#### Recovery validation

5m rate must drop below 0.1/s. Genuinely-clean
browser SDKs should produce ZERO CSRF rejects.

### 3.5. QSDSecurityInvalidTokenSpike

#### What triggered it

`rate(QSD_security_auth_invalid_token_total[5m]) > 1.0`
sustained for 10m.

`AuthMiddleware` rejected presented JWTs as
signature-invalid, expired, or unparseable.

#### Triage

```
# Bucket by failure reason.
jq -r 'select(.event=="auth_invalid_token") | .reason' /var/log/QSD/audit.log \
  | sort | uniq -c | sort -rn

# Reasons:
#   "signature invalid" -> brute-force OR wrong
#                          key on the client.
#   "token expired"     -> stale client deploy.
#   "token revoked"     -> see §3.6.
```

#### Mitigation

- **Mostly "expired":** a deploy rollout left
  one client revision with stale tokens. They
  will refresh and the rate decays — no action.
- **Mostly "signature invalid":** rotate the
  ML-DSA signing key if it has been >90d since
  the last rotation; otherwise treat as
  reconnaissance and rely on the rate-limit to
  hold.

#### Recovery validation

5m rate must drop below 1.0/s. Persistent
"signature invalid" beyond 30m is anomalous —
escalate to §3.7 if the source clients also
sign requests.

---

### 3.6. QSDSecurityTokenRevokedHit

#### What triggered it

`increase(QSD_security_token_revoked_hits_total[15m]) > 0`
sustained for 5m.

`TokenRevocationStore` caught a JWT that its
rightful owner had already logged out. Because
the token's signature was still valid, the
replayer holds a copy of the raw JWT — the
token leaked.

**This is the only `critical`-severity alert in
the group. Page out of band, not just into
Slack.**

#### Triage

```
# Which user's token was replayed?
jq -r 'select(.event=="token_revoked_hit") | .user_id' /var/log/QSD/audit.log

# What IP is replaying?
jq -r 'select(.event=="token_revoked_hit") | .client_ip' /var/log/QSD/audit.log
```

The user_id and the replaying IP are the two
data points the security lead needs to start
the incident review.

#### Mitigation

1. **Notify the affected user immediately** — a
   leaked token means at minimum their session
   was intercepted, at worst their keystore was
   compromised.
2. **Rotate the user's keystore** (force a new
   ML-DSA key generation, invalidate the old
   one).
3. **LB-block the replaying IP.**
4. **Snapshot the audit log for the past 24h**
   filtered on the affected `user_id`. The
   forensic window is short — the audit log
   rotates and the operator must preserve
   evidence.
5. **Open a high-severity incident ticket.**

#### Recovery validation

`increase()` over a fresh 15m window must
return to 0. The original alert may stay sticky
for 15m as the window evicts the captured hit;
that is expected.

### 3.7. QSDSecurityRequestSignatureFailureSpike

#### What triggered it

`rate(QSD_security_request_signature_failed_total[5m]) > 0.1`
sustained for 5m.

`RequestSigningMiddleware` rejected the HMAC
over (method, path, body, ts).

#### Triage

```
# Bucket by failure reason. The middleware
# distinguishes signature mismatch from clock
# skew from a missing header in the audit log.
jq -r 'select(.event=="request_signature_failed") | .reason' /var/log/QSD/audit.log \
  | sort | uniq -c | sort -rn
```

Likely causes, in descending order of
probability:

1. SDK upgrade without the matching secret
   update.
2. Clock skew between client and server >
   `MaxClockSkew` (30s).
3. Active man-in-the-middle tampering with
   bodies of in-flight requests.

#### Mitigation

- **SDK secret rotation drift:** coordinate
  the missed rotation with the SDK consumers.
- **Clock skew:** check the offender's NTP
  state. If it's a node operator, send them
  the operator-guide § on NTP.
- **Tampering:** TLS termination is supposed
  to prevent body tampering. If you see this
  at scale, your TLS edge is broken — page
  the platform operator.

#### Recovery validation

5m rate must drop below 0.1/s.

---

### 3.8. QSDSecurityRequestTimeoutSpike

#### What triggered it

`rate(QSD_security_request_timeout_total[5m]) > 0.5`
sustained for 10m.

`RequestTimeoutMiddleware` cancelled the
handler context after the
`DefaultRequestTimeout` (30s) deadline.

#### Triage

```
# Does a single route dominate?
jq -r 'select(.status==504) | .path' /var/log/QSD/audit.log \
  | sort | uniq -c | sort -rn | head -10

# Compare with handler-latency histograms in
# Grafana — single-route concentration =
# code-path bug; flat = connection-holding
# attacker.
```

#### Mitigation

- **Single route concentration:** open a perf
  ticket for that handler. Roll the offending
  route into a request-timeout bypass list
  (`pkg/api/request_timeout.go::requestTimeoutBypass`)
  ONLY if the route is intentionally
  long-running (e.g. a long-poll); otherwise
  fix the handler.
- **Flat distribution / Slowloris:** reduce
  the upstream LB / proxy
  `proxy_read_timeout` / `proxy_send_timeout`
  to shed slow connections at the edge before
  they consume node workers.

#### Recovery validation

5m rate must drop below 0.5/s.

---

### 3.9. QSDSecurityCORSRejectionSpike

#### What triggered it

`rate(QSD_security_cors_rejections_total[5m]) > 1.0`
sustained for 10m.

`CORSMiddleware` rejected a browser preflight
or actual request because its `Origin` header
did not appear in the
`QSD_CORS_ALLOWED_ORIGINS` allowlist.

#### Triage

```
# Which Origin is being rejected?
jq -r 'select(.event=="cors_reject") | .origin' /var/log/QSD/audit.log \
  | sort | uniq -c | sort -rn | head -10
```

A single repeated value means a missing entry
on the allowlist; a wide distribution is a
scanner that should be ignored.

#### Mitigation

- **Single missing legitimate origin:** add
  the origin to `QSD_CORS_ALLOWED_ORIGINS`
  (comma-separated), restart the API. The
  allowlist is read at process start only.
- **Scanner distribution:** no action — the
  alert is designed to flag at the legitimate
  threshold for a properly-configured fleet.
  If your fleet has many short-lived browser
  origins (e.g. preview deploys), consider
  raising the `for:` window from 10m to 30m.

#### Recovery validation

5m rate must drop below 1.0/s.
